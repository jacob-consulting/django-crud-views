# Internationalization (i18n)

This page describes how to add or update translations for the `django-crud-views` packages and
the example applications.

## Overview

Translations live in each package's `locale/` directory, following Django's standard layout:

```
<package>/locale/<lang>/LC_MESSAGES/django.po   # source strings + translations
<package>/locale/<lang>/LC_MESSAGES/django.mo   # compiled binary (generated)
```

The five packages that ship message catalogs are:

| Package | Locale path |
|---|---|
| `crud_views` | `src/crud_views/locale/` |
| `crud_views_workflow` | `src/crud_views_workflow/locale/` |
| `crud_views_polymorphic` | `src/crud_views_polymorphic/locale/` |
| `crud_views_guardian` | `src/crud_views_guardian/locale/` |
| `crud_views_object_detail` | `src/crud_views_object_detail/locale/` |

Each package ships **seven locales**:

| Language | Locale code | Directory | Status |
|---|---|---|---|
| English | `en` | — (source language, no `.po`) | source strings |
| German | `de` | `de` | human-authored, authoritative |
| French | `fr` | `fr` | machine-authored, pending native review |
| Spanish | `es` | `es` | machine-authored, pending native review |
| Portuguese | `pt` | `pt` | machine-authored, pending native review |
| Italian | `it` | `it` | machine-authored, pending native review |
| Simplified Chinese | `zh-hans` | `zh_Hans` | machine-authored, pending native review |

`en` is the language the source strings (`msgid`s) are written in, so it needs no `.po`/`.mo`
pair — Django falls back to the literal `msgid` when no catalog is active.

Note the locale-code vs. directory-name mismatch for Chinese: the Django/BCP-47 language code
is `zh-hans` (used in `LANGUAGES` and URLs), but gettext requires the on-disk directory to be
`zh_Hans` (used under `locale/`).

The `fr`, `es`, `pt`, `it`, and `zh-hans` catalogs were translated by machine and are **pending
native-speaker review** — each `.po` file carries a `NOTE:` header comment saying so. Only `de`
is human-authored and considered authoritative. See
[issue tracking](#machine-authored-locales) below.

The `examples/bootstrap5` example app has its own `locale/` directory (for app-specific strings
like feature-app labels) and is fully wired up to Django's i18n infrastructure — see
[example-app infrastructure](#example-app-infrastructure).

## Requirements

GNU gettext must be installed:

```bash
# Debian/Ubuntu
sudo apt-get install gettext

# macOS
brew install gettext
```

## Updating translations after code changes

When translatable strings are added or changed in Python or template files, regenerate the
`.po` files and fill in any new entries.

**1. Extract new strings** using the repo-root taskfile tasks, which loop over all five packages
and all six translated locales (`de fr es pt it zh_Hans`):

```bash
task msg-make       # all packages: crud_views, crud_views_workflow, crud_views_polymorphic,
                     # crud_views_guardian, crud_views_object_detail
task examples:msg-make   # example app (examples/bootstrap5)
```

`task msg-make` is defined in the repo-root `taskfile.yaml` and runs, for every package in
`I18N_PACKAGES` and every locale in `LOCALES`:

```bash
cd src/<package>
PYTHONPATH=../.. python -m django makemessages -l <locale> --settings=settings_i18n
```

`task examples:msg-make` delegates to `examples/bootstrap5/taskfile.yaml`, which runs
`manage.py makemessages -l <locale>` for the same six locales from within the example app.

> `settings_i18n.py` (repo root) is a minimal Django settings file used only for management
> commands against the packages — it lists all five packages in `INSTALLED_APPS` so
> `makemessages` can find their templates and Python source.

`makemessages` updates each `.po` file in-place. Existing translations are preserved; new
strings appear with an empty `msgstr ""`; strings that no longer exist are marked with `#~ `
(obsolete).

**2. Edit the `.po` file** and fill in the empty `msgstr` values for the new strings. Remove any
`#, fuzzy` flags after reviewing the suggested translations.

**3. Compile to binary** using the same task pair:

```bash
task msg-comp             # all packages
task examples:msg-comp    # example app
```

`task msg-comp` runs `python -m django compilemessages --settings=settings_i18n` from `src/`,
which compiles every package's catalogs in one pass. `task examples:msg-comp` runs
`manage.py compilemessages` inside `examples/bootstrap5`.

**4. Commit both `.po` and `.mo` files.**

> The `.mo` files are committed so that the packages work out of the box without requiring a
> compile step after installation.

## The language selector

`crud_views` ships a reusable `{% cv_language_selector %}` template tag
(`crud_views.templatetags.crud_views.cv_language_selector`) that renders a CSP-safe Bootstrap
dropdown listing `settings.LANGUAGES`, with a form per language that POSTs to Django's built-in
`set_language` view. Drop it into any template that has a `request` in context:

```django
{% load crud_views %}
{% cv_language_selector %}
```

For the POST to `set_language` to resolve, the consuming project's URLconf must include
Django's i18n URLs:

```python
from django.urls import include, path

urlpatterns = [
    path("i18n/", include("django.conf.urls.i18n")),
    ...
]
```

The dropdown renders nothing (an empty tag) when `settings.LANGUAGES` has only one entry.

## Example-app infrastructure

`examples/bootstrap5` is fully wired up as a working reference for what a project needs. In
`examples/bootstrap5/project/settings.py`:

- `django.middleware.locale.LocaleMiddleware` in `MIDDLEWARE`, after `SessionMiddleware` and
  before `CommonMiddleware`.
- `LANGUAGES` with all seven entries (`en`, `de`, `fr`, `es`, `pt`, `it`, `zh-hans`).
- `LOCALE_PATHS = [BASE_DIR / "locale"]`, pointing at the example app's own `locale/` directory.
- `"django.template.context_processors.i18n"` in the template `context_processors`.

In `examples/bootstrap5/project/urls.py`:

- `path("i18n/", include("django.conf.urls.i18n"))`, which provides the `set_language` view the
  language selector posts to.

The site nav (`examples/bootstrap5/project/templates/project/nav.html`) includes
`{% cv_language_selector %}`, so switching languages is available on every page of the running
example.

## Adding a new language

**1. Add the language to `LANGUAGES`** in your project's settings module — see the
[example-app infrastructure](#example-app-infrastructure) section above for the full set of
settings a project needs for translated content:

```python
from django.utils.translation import gettext_lazy as _

LANGUAGES = [
    ("en", _("English")),
    ("de", _("German")),
    ("nl", _("Dutch")),  # new
]
```

**2. Add the locale code to the taskfile loop vars** — `LOCALES` in the repo-root
`taskfile.yaml` and `examples/bootstrap5/taskfile.yaml` — then run:

```bash
task msg-make
task examples:msg-make
```

This creates `locale/<code>/LC_MESSAGES/django.po` in all five packages and in the example app.
(For a locale whose Django language code differs from its gettext directory name — as with
`zh-hans` → `zh_Hans` — pass the gettext directory name to `makemessages -l`, and use the
language code in `LANGUAGES`.)

**3. Translate all strings** in each new `.po` file.

**4. Compile and commit**:

```bash
task msg-comp
task examples:msg-comp
```

## Machine-authored locales

`fr`, `es`, `pt`, `it`, and `zh-hans` were generated by machine translation to get full
seven-language coverage in place, and are **pending native-speaker review**. `de` is
human-authored and is the only locale currently considered authoritative for tone and
correctness. Each machine-authored `.po` file carries a `NOTE:` header comment recording this.
A native-review follow-up is tracked separately (see `superpowers/specs/` for the drafted
issue).

## Using `gettext_lazy` for class-level strings

Strings defined at class level (model `TextChoices`, `@transition` decorator labels, form class
attributes) must use `gettext_lazy` instead of `gettext`, because they are evaluated at import
time before the request language is known:

```python
# correct
from django.utils.translation import gettext_lazy as _

class Status(models.TextChoices):
    NEW = "new", _("New")
    ACTIVE = "active", _("Active")

# incorrect — evaluated once at import time, ignores active language
from django.utils.translation import gettext as _
```

## Infrastructure checklist

For a Django project to serve translated content, all of the following must be in place
(the example app demonstrates every item — see
[example-app infrastructure](#example-app-infrastructure) above):

- `django.middleware.locale.LocaleMiddleware` in `MIDDLEWARE` (after `SessionMiddleware`, before
  `CommonMiddleware`)
- `LANGUAGES` list in `settings.py`
- `LOCALE_PATHS` pointing to the project's own `locale/` directory
- `"django.template.context_processors.i18n"` in the template context processors
- `path("i18n/", include("django.conf.urls.i18n"))` in `urls.py` (for the language switcher and
  the `{% cv_language_selector %}` tag)
